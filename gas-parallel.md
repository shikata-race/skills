# gas-parallel — 並列エージェントでGASを高速開発するスキル

plan.mdのタスクリストからファイル競合なしで最大4並列実装し、clasp push → deploy まで一気通貫で行う。

## 使い方

```
/gas-parallel
```

カレントディレクトリの `plan.md` と `.clasp.json` を自動検出する。

---

## 実行手順

### STEP 1: 現状把握

- `plan.md` を読んで未完了タスクを列挙
- `.clasp.json` から deploymentId を取得
- プロジェクトの `.gs` ファイル群と `index.html` を glob で一覧化

### STEP 2: タスク選定とファイル割り当て

**並列化の判断基準：タスクが触るファイルが重ならないか**

| ファイルの性質 | 並列化のリスク | 対策 |
|---|---|---|
| エントリーポイント（APIルーター相当） | 高（追記が重なりやすい） | 1エージェントのみ |
| フロントエンド（大きなSPA） | 高（スクロール量が多く誤認しやすい） | 1エージェントのみ |
| 新規ファイル作成 | なし（ファイルが存在しない） | 独立エージェントに最適 |
| マニフェスト・設定ファイル | 低（構造が単純） | 独立エージェントに切り出すと安全 |
| サービス層（機能別に分割済み） | 低（責務が明確） | 担当ファイルが異なれば並列可 |

**割り当てパターン（4並列の典型）：**

```
Agent 1 → 新規ファイル作成（競合ゼロ）
Agent 2 → エントリーポイント（APIルーター）に新関数追記のみ
Agent 3 → 特定サービスファイル（機能別に独立したもの）
Agent 4 → フロントエンド or マニフェスト
```

複数タスクを1エージェントに集約するのは「同一ファイル内のみ」に限定する。

### STEP 3: 各エージェントへの指示作成

プロンプトに必ず含める要素：
- **対象ファイルのパス**（読んでから修正と明示）
- **実装仕様**（曖昧にしない。関数シグネチャ・引数・戻り値まで書く）
- **既存コードとの整合**（呼び出し元が期待する形式・定数名・エラー処理方針）
- **ツール指定**（既存ファイルは `Edit`、新規は `Write`）

### STEP 4: 1メッセージで4エージェント同時起動

`run_in_background: true` で全エージェントをバックグラウンド起動し、完了通知を待つ。

### STEP 5: 完了後の確認

- 各エージェントのサマリーを読む
- 変更が意図通りか `Read` で抜き打ち確認
- ファイルをまたぐ依存関係（関数の戻り値の型変更など）を重点確認

### STEP 6: push → deploy

```bash
clasp push --force
clasp deploy --deploymentId <id>
```

### STEP 7: plan.md に完了マークを追記

## よくある失敗パターンと対策

| 失敗 | 原因 | 対策 |
|---|---|---|
| エージェントが存在しない関数を呼ぶ | 他のエージェントが追加する関数を前提にしていた | 依存関係がある場合は逐次実行に変える |
| 同じファイルの同じ箇所を両方が書き換える | タスク分割が甘い | ファイル単位で担当を完全分離 |
| 大きなファイルで挿入位置を誤る | Read せずに推測で Edit している | プロンプトに「必ずReadしてから」と明記 |
| push後にデプロイが古いバージョンを指す | deploymentId の指定漏れ | `.clasp.json` から毎回確認 |

### 2026-03-25
- **効いたパターン**: マニフェスト（スコープ追加）を独立エージェントに切り出すと他と干渉しない
- **効いたパターン**: 「新規ファイル作成」タスクは競合ゼロなので必ず並列に入れる
- **注意**: エントリーポイントファイルに複数エージェントが追記するとコンフリクト。「末尾追記のみ」でも1エージェントに絞るのが安全
- **注意**: 並列エージェントが互いの戻り値の型変更に依存していると片方が壊れる。インターフェースを先に決めてからエージェントに渡す
- **コツ**: フロントエンドの大きなHTMLファイルはプロンプトに「N行目の〇〇関数の後に追加」と行番号まで明示すると誤挿入が減る
